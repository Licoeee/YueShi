🎨 《悦时 (Yue Shi)》视觉与代码规范

1. 核心视觉参数 (The Soul)

1.1 背景渐变 (Splash & Background)

代码：radial-gradient(circle at 50% 40%, #FFF5F7 0%, #FFD1DC 60%, #FEB47B 120%)

说明：模拟柔和的逆光感，中心偏白粉。

1.2 物理反馈按钮 (Vibrant Button)

常规状态：

background: linear-gradient(135deg, #FF7E5F 0%, #FEB47B 100%)

box-shadow: 0 8px 20px -5px rgba(255, 126, 95, 0.6), inset 0 2px 3px rgba(255, 255, 255, 0.4)

border: 1px solid rgba(255, 255, 255, 0.2)

按下状态 (:active)：

transform: translateY(4rpx) (向下位移，严禁使用 scale)

box-shadow: inset 0 4px 12px rgba(0,0,0,0.15)

欢迎页入场状态：

按钮初始需从中心向两侧展开，并伴随透明度提升与轻微虚化收敛；动画完成后才进入常规可点击状态。

1.3 文字美学

品牌标题：欢迎页主标题使用圆润厚重字体栈 `YouYuan, FZYaoti, "PingFang SC", "Microsoft YaHei", sans-serif`；颜色建议为半通透暖玫瑰色 `rgba(180, 86, 94, 0.72)`；层次只能通过柔和阴影建立，禁止硬描边或实心厚描边。

Slogan：欢迎页副标题文案为"遇见你的专属甜蜜时刻"；颜色为暖棕色，字重需要明显高于常规正文，`letter-spacing` 建议在 `0.16em` 到 `0.18em` 之间，避免字形黏连。

1.4 欢迎页蛋糕主视觉

欢迎页主视觉为居中的卡通鲜奶蛋糕，需贴近真实生日蛋糕结构：

- 顶部为三根蜡烛和火苗；
- 奶油层保持白色或微粉白色；
- 蛋糕胚统一改为淡粉色；
- 奶油上的装饰使用零散分布的圆形彩色糖果，不再使用草莓；
- 整体比例需要在手机首屏中央形成明确视觉焦点，但不能压迫标题和按钮区。

1.5 顾客端蛋糕详情体验

- 顾客首页当前阶段只展示蛋糕商品卡，卡片点击进入详情，不在卡片角落额外放置快捷加购图标。
- 商品详情默认使用"底部上拉半屏层"呈现，顶部需保留一段被压暗的原页面露出区，遮罩颜色应偏暖灰或暖棕，避免纯黑遮挡过硬。
- 半屏详情层只承担展示职责，用户在详情区域内上滑后进入独立详情页；半屏态不显示底部购买操作区。
- 独立详情页继续承接完整详情内容与固定底部操作区：首页、购物车为图标型入口；加入购物车、立即购买为主次分明的实体按钮。
- 独立详情页中的奶油类型文案本轮固定为 `乳脂奶油`、`动物奶油`、`裸蛋糕`，购物车、结算、订单与备注页必须复用同一套文案快照。
- 顾客端关键动作前允许弹出登录引导，但登录引导只作为动作前置，不得打断首页浏览与详情浏览节奏。

2. 动画规范

Float (漂浮)：@keyframes float，上下位移 12px，周期 4s。

Pulse (呼吸)：用于蛋糕蜡烛火苗，周期 1.5s。

Materialize (实体化展开)：用于欢迎页 CTA 按钮，从中心向两侧展开并逐渐清晰，建议时长 360ms 到 480ms。

Reveal (圆形转场)：点击"进入主页"后，以屏幕中心为原点向外扩散显示主页内容，需带柔和 easing，并叠加少量粒子拖尾，避免生硬切页。

Lift Sheet (详情上拉)：用于商品详情半屏层，从底部上移进入，建议时长 280ms 到 360ms，禁止出现生硬的线性位移动画。

Detail Expand (详情展开)：用于半屏详情页进入独立详情页，触发条件为在详情区域内继续上滑；需保留拖拽连续感，避免切成完全无关的第二套动效。

3. 开发约束

适配：必须使用 rpx 或 vw。

组件：UI 优先使用 TDesign；但当 `t-button` 等组件的内建盒模型会持续破坏关键交互区稳定性时，可改用沿用同一暖粉橙视觉 token 的自定义 action shell 承载点击层。禁止无约束原生拼装。

4. 微信小程序实现约束

4.1 长页面渐变

复杂渐变不能直接挂在全局 `page` 背景上，长内容页面必须拆成"固定背景层 + 内容层"结构，否则滚动时会出现背景衔接断层。

4.2 TDesign 图标策略

当前工程稳定基线不依赖 TDesign 字体图标。首屏或关键路径页面应避免把视觉成立建立在 `t-icon` 上；使用带默认 icon 的组件时，如果该图标不是必须项，应显式关闭对应 icon 属性。

4.3 微信开发者工具兼容基线

本项目以微信开发者工具 `2.01.2510280` 为校验基线。引入或升级 `miniprogram_npm` 依赖后，必须确认 `App/project.config.json` 中 `setting.es6 = true`，否则真机调试可能无法解析依赖产物。

4.4 第三方包补丁

若为 TDesign 做兼容性补丁，必须同时修改源包和生成包对应文件；只改一侧会在"构建 npm"后丢失修复结果。

4.5 欢迎页动画策略

欢迎页核心动效优先使用 WXML/WXSS/TS 状态驱动实现，不再把 Lottie 作为首屏品牌动画的主路径依赖。

4.6 商品详情状态一致性

半屏详情态与独立详情页必须共享同一份商品内容上下文；独立详情页独占规格状态与底部操作语义，禁止出现两套详情信息割裂或按钮行为不一致的情况。

4.7 TDesign Button CSS 变量定制规范

当页面继续使用 `<t-button>` 时，优先通过 `t-class` 在外部传入自定义类名，再在该类名上设置 TDesign CSS 变量。禁止使用 `.t-button--outline` 等内部选择器在组件外部强行覆盖。

若根因分析确认问题不在配色变量，而在 `t-button` 的默认高度、圆角、边框语义、内容压缩策略或内部盒模型，则不应继续把“调按钮变量”当成唯一解法；此时允许切换为自定义 action shell，并沿用同一视觉层级、颜色、阴影和禁用态规范。

各 variant 对应的变量前缀映射如下（以 `theme="default"` 为例）：

| variant | 颜色 | 背景 | 边框 |
|---|---|---|---|
| （无/default） | `--td-button-default-color` | `--td-button-default-bg-color` | `--td-button-default-border-color` |
| `outline` | `--td-button-default-outline-color` | `--td-bg-color-container`（容器级） | `--td-button-default-outline-border-color` |
| `text` | `--td-button-default-color` | 透明 | 无 |

`theme="primary"` 时前缀替换为 `--td-button-primary-*`，`variant="outline"` 时为 `--td-button-primary-outline-*`，余类推。

定制前必须在 `miniprogram_npm/tdesign-miniprogram/button/button.wxss` 中搜索确认变量名，不得凭记忆推断。

4.8 详情页底栏布局规范

底部固定操作栏采用"导航入口 + 操作按钮"两组结构时，布局方案如下：

- **整体容器**：使用 `display: flex; align-items: stretch` 而非固定列宽的 `grid`，避免不同屏宽下溢出。
- **导航入口**：使用纯 `<view>` 实现（非 `<t-button>`），设置固定宽高，`flex: none`。
- **操作按钮组**：使用 `<t-button>` + `flex: 1`，让操作区自适应剩余空间。
- **高度统一**：导航格子与操作按钮必须显式设置相同高度值。

4.9 顾客端高风险动作区布局契约

购物车底栏、结算底栏、订单操作区这类高风险区域，必须先定义“固定动作区 + 弹性内容区”的布局契约，再做视觉微调。

- **购物车底栏基线**：优先使用 `grid-template-columns: minmax(0, 1fr) 固定 CTA 宽度`；左侧主区再拆成“固定切换按钮宽度 + 弹性金额文案区”。
- **金额区约束**：金额区必须 `min-width: 0`，且只能占剩余空间；禁止让金额文本把“去结算”或“提交订单”按钮挤走。
- **同行规则**：`合计`、货币符号和金额必须保持同一横排，推荐 `white-space: nowrap`，并让标签区 `flex: none`、金额区 `flex: 1`。

4.10 长金额收敛策略

长金额的第一解决手段是稳定布局，不是强行截断。

- **优先级**：先固定 CTA 宽度，再让金额区占剩余空间，最后才使用动态字号收敛。
- **动态字号**：当金额达到四位数、五位数等极限场景时，允许切换 `tight` / `compact` 字号类，但仍需保持金额完整可读。
- **禁区**：不要通过把金额拆行、纵排、插入省略点或缩到不可读来规避布局问题。

4.11 Action Shell 规范

顾客端当前稳定基线中的高频动作区，允许用自定义 action shell 统一承载视觉和点击反馈。

- **适用区域**：购物车勾选 pill、购物车全选/结算按钮、结算页历史手机号入口与提交按钮、备注页操作按钮、我的页操作按钮、设置页清理与管理按钮。
- **视觉层级**：统一保留暖粉橙渐变主按钮、浅底描边次按钮、淡红危险按钮、灰粉禁用按钮四个层级。
- **盒模型基线**：显式设置 `min-height`、横向 `padding`、圆角、边框与阴影；禁止依赖内容本身撑出按钮边界。
- **交互要求**：禁用态必须显式降低对比度并阻断点击，危险态不能只靠文字变红。

4.12 辅助入口解耦规范

辅助入口不得复用主输入框或主操作按钮本体。

- **历史手机号**：必须作为独立信息卡或独立触发入口出现，并配合 `ActionSheet` / 弹层选择，不再压缩手机号输入框本身。
- **选择结果回填**：辅助入口只负责触发和回填，主输入框仍保留手动编辑能力。

4.13 状态闭环规范

提示文案、按钮禁用态、点击拦截必须来源于同一份状态。

- **取货时间错误**：当选择到早于当前时间的月份/日期/时段时，页面内要显示明确 warning，提交按钮同步变为不可点击。
- **提交拦截**：若用户仍尝试点击提交逻辑，错误文案应与当前 warning 同源，禁止出现两套冲突提示。

4.14 长文本承载规范

账号名、订单号、备注、长规格与其他极限内容，必须进入可容纳长文本的结构。

- **账号信息**：长账号名优先用独立信息卡承载，并设置 `min-width: 0` 与合理换行策略；不要继续塞进 `t-cell note`。
- **订单号**：订单号与标签需要保持在同一行时，应让标签固定宽度、值区允许省略，避免直接换行打断阅读。
- **商品快照**：购物车、结算页、订单页、备注页的商品信息优先显示快照字段和封面图，不再回列表反查。

4.15 购物车与订单视觉补充规范

- **购物车选中态**：禁止只靠细边框小方框表达勾选。必须采用“卡片高亮 + 清晰勾选控件”的组合，选中卡片可使用淡粉橙背景、暖色描边与轻量 glow，但不得偏离现有暖粉橙体系。
- **购物车封面图**：购物车、结算页、订单列表、订单详情统一显示商品首页封面图；图片比例优先保持接近首页商品卡主图，避免裁切变形。
- **购物车底部汇总**：`合计` 与价格必须同一行，价格始终位于 `合计` 右侧；“去结算”按钮固定在右侧，不允许因金额位数变化发生横向偏移。
- **左滑删除**：购物车删除入口使用 TDesign `swipe-cell`，删除按钮采用暖橙偏危险态，不使用生硬纯黑/纯灰块；滑动回弹与关闭交互必须自然。
- **订单状态 Tab**：顾客订单页状态切换采用圆角胶囊式分段视觉，活动态强调暖色填充与清晰文字对比，非激活态保持轻量玻璃感，不使用廉价标签堆叠感。
- **备注页按钮**：`编辑备注`、`取消订单`、`保存备注`、`取消` 必须使用具备明确边界的主次按钮形态；当 `t-button` 盒模型无法稳定承载时，可切换为沿用同一视觉 token 的 action shell，禁止再用纯文字链接充当主要操作。
- **订单号排版**：订单详情中的订单号必须保持单行显示；若空间不足，优先压缩次级信息，不允许订单号本体换行。
- **顾客“我的”页**：用户头像、昵称、登录状态区建议采用单张主卡承载；设置入口与清理入口采用列表型布局，保持信息层级清晰，不堆砌无关功能。

4.16 顾客端界面稳定性规范

- **先定布局契约，再做视觉微调**：购物车底栏、结算页表单区、备注/设置/我的页操作区，必须先定义“固定宽度区域、弹性剩余区域、允许收缩的文字区域”，再进入字号、配色、阴影调整。禁止直接用 `justify-content: space-between` 硬撑所有动态内容。
- **金额与 CTA 的稳定布局**：购物车、结算页底栏若右侧存在主 CTA，CTA 区必须固定宽度；金额区只能占剩余空间，并显式设置 `min-width: 0`。金额超过常规位数时，优先动态缩字号，其次截断次级文案，禁止遮挡按钮。
- **输入区不承载次级入口**：手机号输入框、备注输入框等主输入组件不得再承担历史记录入口、辅助选择入口等次级动作。历史手机号、快捷选择等能力必须独立成一行或独立卡片，避免压缩输入内容。
- **长文本承载方式**：长昵称、长订单号、长手机号历史项，不允许继续塞进 `t-cell note` 这类天然窄槽位；必须使用独立信息卡、单独文案块或可控换行区承载。
- **按钮可见性排查顺序**：遇到“按钮像文字、边界看不见”时，必须依次排查：组件语义是否选错、TDesign 变体变量前缀是否正确、容器背景是否吞边界、按钮高度/内边距是否塌缩、`::after` 边框是否被覆盖。禁止只通过“加深颜色”粗暴处理。
- **警告态必须联动交互态**：例如取货时间选择错误，不仅要出现提示条，还必须让“提交订单”这类关键动作同步进入禁用态，并由同一个状态字段驱动，不能视觉和行为分离。
- **滑动删除需同时交付结构与动效**：购物车左滑删除不只是“出现删除字样”，必须同时保证手势响应层、删除槽位宽度、关闭回弹、点击删除触发链路四者齐全。
- **浏览器稿到小程序稿的翻译规则**：浏览器设计稿里任何“固定按钮 / 自适应文案 / 极限内容缩放”策略，落地小程序时都要映射成显式类名和状态字段，不允许只凭肉眼照着抄样式。

4.17 界面修复验证清单

- **极限数据**：至少覆盖 `¥1290` 及以上金额、11 位手机号、长账号昵称、长订单号、空态与满态。
- **状态切换**：至少覆盖未选 / 已选 / 全选、可提交 / 不可提交、登录 / 未登录、可取消 / 不可取消。
- **结构核对**：确认关键布局节点是否真的拆成固定区、弹性区和辅助区；只看视觉截图不算完成。
- **测试同步**：设计和结构变化后，要确认测试断言没有继续依赖旧节点名、旧属性或旧组件语义。

4.18 Scene 模式与 Note 防截断口径

- **Scene 购物车删除链路**：凡是通过 `role-page-scene` 承载顾客购物车场景的宿主页（如首页/订单/我的），都必须保留页面级删除弹窗状态（`deleteDialogVisible`、`pendingDeleteItemId`、`cartRefreshTick`）并处理 `request-cart-delete`。禁止只保留左滑手势，不接删除确认与刷新链路。
- **`t-cell note` 专用样式**：我的页、设置页中用于状态短文案的 `note` 不得复用 `app-interactive-text` 这类默认省略类，必须使用专用 note 类并显式设置 `flex: none`、`min-width`、`white-space: nowrap`、`overflow: visible`、`text-overflow: clip`，避免被 `description` 区挤压截断。
- **断言要求**：上述两类规则必须有对应自动化断言，至少覆盖“场景页删除链路存在”和“note 使用专用非截断类”。

4.19 TDesign 组件注册口径

- **WXML 与 JSON 必须成对维护**：凡页面/组件 WXML 中新增 `t-dialog`、`t-swipe-cell` 等 `t-*` 组件，必须同步在对应 `.json` 的 `usingComponents` 注册。禁止只改 WXML/TS 不改 JSON。
- **异常判定信号**：若出现“事件已触发、状态已更新，但界面无反应”，优先检查目标组件是否漏注册，而不是先怀疑业务状态链路。
- **回归断言**：场景宿主页（`home/orders/profile`）涉及删除或确认弹层时，测试需同时断言 WXML 存在实际使用的弹层组件（如 `<t-popup>` 或 `<t-dialog>`）且对应 JSON 注册完成，防止“状态已更新但界面无反馈”的回归。

4.20 测试绿但功能未实现防复发口径

- **完成定义强制双轨**：关键路径修复必须同时满足“自动化测试通过”和“微信开发者工具 2.01.2510280 人工回归通过”；仅满足其一，均不得标记为完成。
- **证据链最小要求**：每条关键路径回归必须记录 `操作步骤 + 预期效果 + 实际结果`；没有这三项证据，不允许输出“已修复”结论。
- **断言设计红线**：测试不能只验证静态结构或字符串存在，必须覆盖至少两层：行为可达（事件触发/状态更新）与用户可见反馈（UI 变化/弹窗展示/按钮状态变化）。
- **冲突裁决规则**：若出现“测试通过但真实交互失败”，以真实交互失败为准，立即回到定位链路，禁止以测试绿灯推进提测、合并或结案。
- **本次典型反例**：删除链路日志显示事件已到达且状态已更新，但因 `t-dialog` 漏注册导致 UI 无反馈；此类问题必须优先检查“组件注册与渲染链路”而非仅看测试结果。

4.21 商家端主题弹窗规范

- **统一承载层**：商家端高风险确认与删除操作，优先使用 `t-popup + 主题 dialog shell`，禁止再混入默认系统感强的原始弹窗风格。
- **适用范围**：订单状态流转确认、订单删除、商品删除、商品回收站永久删除、默认价格项删除等商家端关键确认链路，必须复用同一套主题弹层语言。
- **结构要求**：弹层固定为“标题 + 说明文案 + 操作按钮”三段；标题和说明文案均居中；说明文案最多承担风险说明，不堆辅助小字。
- **视觉要求**：承托层使用暖粉橙浅底渐变、24rpx 以上圆角、同主题 glow 阴影；禁止纯白底 + 细灰边的默认样式。
- **按钮层级**：取消按钮使用 `merchant-button-token--outline`；确认按钮按语义使用 `merchant-button-token--primary` 或 `merchant-button-token--danger`；危险操作不能只把文字改红。
- **实现口径**：若页面已经使用 `t-popup` 主题弹层，则应移除无用的 `t-dialog` 注册和旧状态字段，避免出现两套删除弹窗并存。
- **默认价格保存口径**：默认价格属于跨页面价格基线，禁止输入即写盘。商家端必须采用“本地草稿编辑 + 明确保存按钮”的模式；未点击保存前，新增项、删除项、价格修改都只存在于当前草稿态。
