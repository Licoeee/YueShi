# 商家端多图商品、订单详情与定时清理回收站设计

## 背景

本轮目标是在不破坏既有商家端能力的前提下，继续完善以下链路：

- 商品编辑规格扩充、多选奶油、多图上传与回填
- 搜索框清空按钮视觉统一
- 商家订单详情查看、字段放大与字体切换
- 已完成订单的定时清理、立即清空与回收站链路
- 订单日期选择窗口主题统一

约束如下：

- 全程遵守 `AGENTS.md`、`docs/STYLE_GUIDE.md`、`docs/PRD.md`、`docs/TODO.md`
- 代码改动前先完成本设计与实施计划
- UI 优先使用 TDesign；若关键布局不能稳定承载，再保留已有 action shell 例外
- TypeScript 严格模式，禁止 `any`
- 不修改无关模块，不提前更新 `docs/TODO.md`

## 现状结论

### 1. 商品数据结构

当前商家商品模型仍是单值结构：

- `specSizes: ProductSpecSize[]`
- `creamType: MerchantCreamType`
- `coverImage: string`

这意味着：

- 规格目前只能使用 `6/8/10-inch`
- 奶油类型不能多选
- 图片只能保存一张
- 编辑回填、列表展示、批量编辑、搜索都只能处理单奶油和单图

### 2. 搜索框实现分裂

当前搜索输入存在两类组件：

- 顾客端首页主要使用 `t-search`
- 商家页主要使用 `t-input clearable`

两者的内部 DOM 结构和类名不同，因此清空按钮若要统一成圆形，不能只改页面样式，必须从 TDesign 组件主题层做统一补丁。

### 3. 订单删除链路

当前商家端已完成订单删除是物理删除：

- `deleteCompletedMerchantOrder()` 直接从订单快照移除
- 自动清理也是直接过滤完成订单

这不满足“立即清空 -> 进入回收站 -> 可查看/恢复/彻底清空”的需求，因此必须先把订单删除链路改成“进入商家回收站”的状态流，而不是继续真删除。

### 4. 订单详情能力缺失

当前订单管理页只有卡片列表，没有商家订单详情页，也没有字段级放大弹窗。因此这次新增能力必须同时覆盖：

- 页面路由
- 详情结构
- 放大弹窗状态
- 字号切换
- 长文本和大图滚动

### 5. 日期弹窗主题只调了高度

当前订单日历仅处理了 `panel-height`，并没有把头部、选中态、禁用态、确认区和安全区接入项目主题，导致视觉风格仍与商家端不一致。

## 方案对比

### 方案 A：兼容扩展层，最小破坏，推荐

做法：

- 在商家商品模型中新增 `creamTypes` 和 `imageUrls`
- 保留 `creamType` 与 `coverImage` 作为兼容字段，并由新字段派生
- 订单新增 `merchantRecycleMeta`，让手动删除、立即清空、定时清理共用“移入商家回收站”链路
- 订单详情单独建页，放大查看通过字段配置驱动

优点：

- 兼容旧快照和当前列表逻辑
- 风险集中，可逐步落地
- 对现有顾客端和商家端结构破坏最小

缺点：

- 过渡期会同时存在旧字段与新字段

### 方案 B：商家模型整体重置

做法：

- 直接把商品模型改成纯数组结构，移除旧字段
- 订单删除与回收站完全拆成新存储层

优点：

- 模型最干净

缺点：

- 迁移成本高
- 与当前已存在的商家功能冲突面大
- 不符合本轮“最小改动、不破坏现有功能”的边界

### 方案 C：只改 UI 层承载

做法：

- UI 上支持多图和多选，但存储仍塞回字符串或首图字段
- 回收站只做展示层，不改删除逻辑

优点：

- 表面改动快

缺点：

- 回填、搜索、恢复、预览会持续出现不一致
- 属于表层补丁，不可接受

## 推荐方案

采用方案 A。

理由：

- 可以在当前商家端基线上稳定演进
- 商品、搜索、订单详情、回收站、日历主题都能落到明确的数据流上
- 能兼容已经做过的商品 ID、瀑布流、批量编辑、订单状态流转等能力

## 设计细化

### 1. 商品模型与编辑器

#### 数据结构

- `ProductSpecSize` 扩展为：`6/8/10/12/14/16-inch`
- `MerchantCreamType` 扩展为：
  - `animal-cream-i`
  - `dairy-cream`
  - `naked-cake`
- `MerchantProductRecord` 新增：
  - `creamTypes: MerchantCreamType[]`
  - `imageUrls: string[]`
- 兼容字段保留：
  - `creamType` 取 `creamTypes[0]` 的回退值
  - `coverImage` 取 `imageUrls[0]` 的回退值

#### 编辑器交互

商品编辑弹窗改为：

- 规格多选
- 奶油多选
- 多图上传
- 缩略图列表
- 单张删除
- 设为首图
- 点击缩略图预览大图

#### 搜索兼容

商品搜索字段继续覆盖：

- `id`
- `title`
- `description`
- `specSummary`
- `creamSummary`

其中 `creamSummary` 由多选奶油类型拼接生成。

### 2. 搜索框清空按钮统一

统一策略：

- `t-search` 和 `t-input clearable` 都改为圆形清空按钮
- 不在页面里单独补丁，直接调整组件主题样式
- 所有补丁必须同步到：
  - `miniprogram/miniprogram_npm/tdesign-miniprogram/...`
  - `miniprogram/node_modules/tdesign-miniprogram/miniprogram_dist/...`

目标：

- 圆形背景
- 中心对齐
- 热区足够
- 不遮挡文本
- 手机窄宽度下仍稳定

### 3. 多图上传与预览

上传链路设计：

- `wx.chooseMedia({ count: 9 })`
- 选择结果追加到 `imageUrls`
- 列表首图即商品卡封面

预览链路设计：

- 使用 `wx.previewImage({ current, urls })`
- 自动支持左右滑动
- 编辑时回填全部图片
- 删除单图后若删掉首图，自动以前一张或下一张补位为新首图
- 调整顺序先采用“设为首图”最小实现，不在本轮引入拖拽排序

### 4. 商家订单详情与字段放大

#### 页面结构

新增：

- `pages/merchant/order-detail`

详情页包含：

- 订单头部摘要
- 商品图片与商品清单
- 取货时间
- 联系人
- 备注
- 规格/奶油/数量
- 金额
- 下单时间

#### 放大交互

每个关键字段右侧增加放大入口：

- 图片：显示大图预览内容
- 文本字段：弹窗展示完整内容
- 长规格或长备注：支持内部滚动

弹窗能力：

- 字号切换：小 / 中 / 大
- 弹窗内部滚动
- 图片字段与文本字段共用一个弹窗壳层，不共用一份静态内容

### 5. 定时清理与订单回收站

#### 数据结构

给 `OrderRecord` 新增：

- `merchantRecycleMeta?: { deletedAt: string; source: 'manual' | 'auto-cleanup' | 'instant-clear' }`

#### 链路规则

- 手动删除已完成订单：进入商家回收站
- 立即清空：把当前所有已完成订单批量移入商家回收站
- 定时清理：到期后自动移入商家回收站
- 回收站支持：
  - 查看
  - 恢复
  - 单条彻底删除
  - 清空回收站

#### 入口设计

- “定时自动清空记录”改名为“定时清理”
- 定时清理弹窗新增“立即清空”按钮
- 弹窗说明区增加“回收站”按钮
- 新增订单回收站页面或弹窗视图，优先独立页面

#### 删除链路原则

- 手动删除、立即清空、定时清理共用同一套移入回收站 helper
- 彻底清空才是真删除

### 6. 日期弹窗主题统一

优先通过 TDesign Calendar 变量接入商家端暖色主题：

- 标题文字
- 关闭按钮
- 星期栏
- 默认日期底色
- 选中日期高亮
- 禁用日期颜色
- 确认按钮区
- 底部安全区背景

如果变量不足，再做最小组件样式补丁，但不随意覆盖无关结构。

自动化测试优先覆盖：

- 商品存储迁移与回填
- 多选奶油与多图保存

- 订单详情页结构与放大弹窗结构
- 搜索框统一样式对应的结构断言

微信开发者工具补测必须覆盖：

- 多图上传与左右滑动预览

- 定时清理与回收站完整链路
- 日期弹窗主题与安全区显示

## 风险与边界

- 顾客端商品目录当前仍是静态 mock，不在本轮改成商家动态商品源。
- 商家端文案按需求保留 `动物奶油i`，不擅自更正为其他词。
- 若 TDesign 搜索/日历主题补丁需要动第三方产物，必须双改 `node_modules` 与 `miniprogram_npm`，否则构建 npm 后会回退。
- 本轮不更新 `docs/TODO.md`，待大帅确认功能完成后再同步。
