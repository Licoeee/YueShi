# 商家商品与订单工作流调整设计

## 背景

本轮目标不是继续做表层样式补丁，而是在既定 V1 视觉方向下，重新校准商家端商品管理页与订单管理页的：

- 信息架构
- 操作阶段表达
- 搜索 / 筛选 / 状态流转的数据链路
- 主要按钮的统一视觉层级

约束如下：

- 不改动与本轮无关的模块
- 继续优先使用 TDesign；若关键交互区已验证不适合 `t-button` 盒模型，才保留既有 action shell 例外
- TypeScript 严格模式，禁止 `any`
- 所有状态流转、删除、自动清理、搜索能力，先复用现有仓储和本地持久化，再补最小新增 helper

## 现状结论

### 商品管理页

- 顶部操作区仍是单行混排，操作阶段不清楚
- `batchEditMode` 只有布尔态，无法区分“正在选商品”和“已经进入批量修改”
- 当前没有搜索字段
- 商品瀑布流本身已稳定，但顶部状态表达与交互流程不符合最新要求

### 订单管理页

- 现在只支持日期筛选，没有状态 Tab、搜索、自动清理入口
- 列表数据只有 `orders` / `visibleOrders` 两层，不足以承载“日期 + 搜索 + 状态 Tab + 自动清理”的叠加
- 商家侧缺少正式的订单状态流转 helper，页面直接扩展会让逻辑分散

### 按钮系统

- 商家端多个页面各自维护 `.merchant-button-token`，视觉方向接近，但仍然分散
- 顾客端已经有稳定的高光渐变按钮风格，可作为 merchant 主按钮 / 次按钮基线
- 商品卡底部操作区已经验证过：继续用 `t-button` 会更容易被双列卡片约束破坏，因此保留 action shell 是合理例外

## 方案对比

### 方案 A：继续在现有页面里打补丁

- 做法：保留当前布尔状态和 `visibleOrders`，用更多字段分别拼接搜索、Tab、清理逻辑
- 优点：改动面最小
- 缺点：状态来源分散，后续极易出现“按钮可点但状态不对”“搜索切换后列表回退”的问题

### 方案 B：重建商品页状态机 + 订单页过滤管线

- 做法：商品页引入明确阶段态；订单页引入统一过滤函数和商家订单管理 helper
- 优点：状态来源单一，便于测试；对现有 UI 结构改动最小；能同时承接搜索、状态流转、自动清理
- 缺点：需要补一层商家订单管理工具文件，并调整现有 scene 测试

### 方案 C：直接把商家端拆成更多子页面

- 做法：把批量选择、批量修改、自动清理设置全部拆出去
- 优点：每页职责更纯
- 缺点：超出本轮最小改动边界，且会引入更多路由和导航成本

## 推荐方案

采用方案 B。

理由：

- 能在不破坏现有 merchant scene 承载方式的前提下，把当前最混乱的状态逻辑收拢
- 商品页只新增“阶段状态机 + 搜索”，不会伤到已稳定的瀑布流与卡片结构
- 订单页把“自动清理 -> 日期筛选 -> 搜索 -> 状态 Tab”固定为同一条数据管线，避免多处 `setData` 互相覆盖

## 设计细化

### 1. 商品页顶部操作区与阶段状态机

商品页顶部改为两行：

- 第一行：`新增商品`、`回收站`
- 第二行：`批量编辑`、`退出编辑`

阶段态改为：

- `idle`：默认浏览态
- `selecting`：已进入商品选择态
- `editing`：批量修改弹窗已打开

交互规则：

- 第一次点“批量编辑”：`idle -> selecting`
- 在 `selecting` 态再次点“批量编辑`：
  - 已选商品 > 0：打开批量修改弹窗，进入 `editing`
  - 已选商品 = 0：toast 明确提示，不进入修改界面
- 点“退出编辑”：无论在 `selecting` 还是 `editing`，都回到 `idle`，并清空本次选择

### 2. 商品搜索

搜索范围先覆盖当前商品列表，字段采用：

- `id`
- `title`
- `description`
- `specSummary`
- `creamLabel`

数据流：

- `activeProducts`
- `productSearchKeyword`
- `filteredProducts`
- `leftColumnProducts` / `rightColumnProducts`

这样不会破坏当前瀑布流分列算法。

### 3. 按钮系统统一策略

统一保留两类主要实现：

- 常规页内主按钮 / 次按钮：继续使用 `t-button + merchant button token`
- 商品卡底部动作区：继续使用 action shell，不回退到 `t-button`

共享 token 规则：

- 主按钮：暖橙渐变 + 内高光 + 清晰描边
- 次按钮：浅底暖边界 + 内高光 + 非纯白容器底
- 危险按钮：浅红底 + 明确边框
- 禁用态：低饱和灰粉 + 明确不可点击

共享样式不放进无关全局页面，而是抽到 merchant scene 可复用样式段中，避免扩大影响范围。

### 4. 订单页过滤与状态流转

订单页新增：

- 搜索框
- 状态 Tab：`待确认 / 待制作 / 待取货 / 已完成`
- 自动清理设置入口

统一过滤顺序：

1. 原始订单快照
2. 自动清理已过期 completed 订单
3. 日期筛选
4. 搜索筛选
5. 状态 Tab 筛选

状态流转规则：

- `pending-payment` / `paid` 显示在“待确认”，动作按钮：`确认付款`
- `in-production` 显示在“待制作”，动作按钮：`制作完成`
- `ready-for-pickup` 显示在“待取货”，动作按钮：`已取货`
- `completed` 显示在“已完成”，动作按钮：`删除订单`

更新状态时必须同步刷新 `updatedAt`，这样自动清理可以把 `updatedAt` 视为“进入 completed 的时间近似值”。

### 5. 自动清理设计

新增独立配置存储键，不塞进订单记录：

- `merchant-order-auto-cleanup-days-v1`

能力设计：

- 允许配置“关闭 / 3 天 / 7 天 / 15 天 / 30 天”
- 触发时机：
  - 订单页进入时
  - 切换 Tab / 日期 / 搜索前
  - 修改自动清理配置后立即重跑
- 删除条件：
  - `status === 'completed'`
  - `now - updatedAt >= configuredDays`

手动删除与自动清理共用同一条“删除已完成订单” helper，避免两套链路。

## 风险与边界

- 当前订单数据仍与顾客端本地订单快照共用；商家删除 completed 订单后，顾客端对应记录也会消失。这符合当前“本地前端闭环”模式，但需要在交付说明里明确这是现阶段架构边界。
- `paid` 当前仍映射到“待确认”聚合态，避免已有脏数据掉出视图。
- 本轮不更新 `docs/TODO.md`，等大帅确认后再同步文档状态。
