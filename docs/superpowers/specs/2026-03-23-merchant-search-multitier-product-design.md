# 商家搜索统一与多层级商品配置设计

## 背景

本轮目标聚焦两类真实问题：

- 搜索框交互存在双清除控件、清除后列表不恢复、三页行为不一致
- 商家商品新增/编辑时，单层、双层、三层虽然在存储层是数组，但页面配置模型仍然是全局扁平结构，无法准确表达“同一商品同时支持多个层级并分别定价”

约束如下：

- 必须遵守 `AGENTS.md`、`docs/STYLE_GUIDE.md`、`docs/PRD.md`、`docs/TODO.md`
- 本轮只修商家端与顾客首页搜索，不直接修改顾客端详情页和选购逻辑
- 若商家端结构调整影响顾客端，必须在完成后补充 `docs/TODO.md` 后续改造项
- TypeScript 保持严格模式，禁止 `any`
- UI 继续优先使用 TDesign；若 TDesign 内建清除图标链路不符合项目约束，可在不破坏整体视觉语言前提下使用自定义清除壳层

## 现状结论

### 1. 搜索框实现已经分裂

当前存在三套使用方式：

- 顾客首页 `customer-home-scene` 使用 `t-search`
- 商家订单页 `merchant-orders-scene` 使用 `t-input`
- 商家商品页 `merchant-products-scene` 使用 `t-input`

这意味着：

- 不能只打一层样式补丁就期望三页行为统一
- 清除按钮、事件触发、值回写本身就来自两套不同组件内部机制

### 2. 双清除控件不是页面手写了两个按钮

根因来自 TDesign 内部清除图标链路：

- `t-search` 内部固定渲染 `t-icon name="close-circle-filled"`
- `t-input clearable` 内部也会通过 `_clearIcon` 渲染默认清除图标

当前项目只给清除容器补了圆形外壳，但没有真正替换内部清除图标来源，因此在小程序里会出现“圆圈外观已经改了，但内部仍残留旧图标/旧占位”的现象。

这与项目红线一致：当前稳定基线不应继续依赖 TDesign 字体图标作为关键交互表现。

### 3. 搜索清除后列表不恢复不是同一个原因

#### 顾客首页

顾客首页 `handleSearchChange()` 里 `setData({ keyword })` 后立刻调用 `syncFeed()`，而 `syncFeed()` 再读取 `this.data.keyword`。清空时会有状态读取时序问题，导致过滤仍然使用旧关键字。

#### 商家订单页 / 商品页

两个页面都只监听了 `bind:change`，没有监听 `bind:clear`。而 `t-input` 内部点击清除时只会触发 `clear`，不会自动走当前页面的 `change` 处理，因此会出现：

- 输入框看起来已清空
- 真正用于过滤的关键字状态没被置空
- 列表没有重算，仍停在旧过滤结果

### 4. 商品层级问题的真正根因是表单模型错误，不是存储层不支持数组

当前商品存储层已经是：

- `layers: MerchantProductLayer[]`
- `priceAdjustmentsByConfigId`
- 默认价格中心按 `single / double / triple` 管理配置项

但商品编辑页仍使用扁平模型：

- `productSpecSizes` 一组
- `productLayers` 一组
- `productCreamTypes` 一组
- `productPricingGroups` 只是把“兼容项”平铺出来

这套模型无法表达：

- 单层支持哪些规格和奶油
- 双层支持哪些组合
- 三层支持哪些组合
- 同一商品中不同层级的配置相互独立

所以当一个商品想同时支持单层和双层时，页面虽然能勾多个层级，但规格、奶油和价目项会互相污染，业务上仍然接近“被迫三选一”。

## 方案对比

### 方案 A：继续修补现有三套搜索和扁平商品表单

做法：

- 三个页面分别补 `bind:clear`
- 分别修样式，把旧清除按钮尽量盖住
- 商品层级继续保留全局规格/奶油/层级，只放宽多选

优点：

- 改动最小
- 短期回归面较窄

缺点：

- 搜索仍有三套实现，后续还会继续分叉
- 商品层级问题只是在 UI 上“看起来可以多选”，数据表达仍然不准确
- 后续顾客端承接时仍然没有稳定的数据边界

### 方案 B：统一搜索壳层 + 商品按层级配置可售价目项

做法：

- 三个页面统一走受控搜索壳层
- 不再使用 TDesign 内建 `clearable` 作为最终清除交互
- 商品编辑页从“扁平筛选”改成“先勾层级，再在各层级下独立勾选价目项并填写加价”

优点：

- 一次解决双清除控件、清空不恢复、行为分裂三类问题
- 商品配置模型与默认价格体系语义对齐
- 老数据仍可兼容回填，不需要本轮直接动顾客端

缺点：

- 商品编辑表单会有一轮结构调整
- 需要补更多回填与迁移测试

### 方案 C：直接重构成顾客端与商家端统一新商品模型

做法：

- 商家商品、顾客详情、购物车、订单快照一起升级成“层级 -> 规格组合 -> 奶油 -> 最终价”模型

优点：

- 模型最完整

缺点：

- 超出本轮范围
- 会直接影响顾客端选购、价格、订单快照
- 不符合“本轮顾客端先不改”的边界

## 推荐方案

采用方案 B。

理由：

- 在不扩大到顾客端重构的前提下，能把本轮根因真正修掉
- 可以复用现有默认价格中心与 `configId` 加价能力，不必重建仓储
- 对现有页面与数据的破坏范围可控，且便于测试

## 设计细化

### 1. 搜索统一设计

#### 目标页面

- 顾客首页
- 商家订单管理
- 商家商品管理

#### 统一原则

- 统一为受控输入
- 统一只保留一个圆形清除按钮
- 统一点击清除后立即恢复完整列表
- 统一支持“输入 -> 清除 -> 再次输入”完整链路

#### 实现策略

- 新增共享搜索壳层组件或共享搜索片段逻辑
- 所有页面不再依赖 TDesign 内建 `clearable` 作为最终可视清除按钮
- 改为：
  - `clearable="{{false}}"`
  - 页面自绘圆形清除按钮
  - 点击后显式调用页面自己的 `handleClearSearch`

#### 这样设计的原因

- 可以真正移除旧清除图标来源，而不是继续在其外面再包一层
- 可以绕过 `t-search` 与 `t-input` 不一致的 `clear` 事件链路
- 可以统一热区、尺寸、对齐、禁用和可视状态

#### 数据流

每页统一采用：

- `keyword`
- `applyFilter(keyword)`
- `clearKeyword()`

而不是“`setData` 后再依赖 `this.data` 读回旧状态”。

### 2. 商品层级配置设计

#### 保留现有基础字段

保留：

- `layers`
- `priceAdjustmentsByConfigId`
- 默认价格快照结构

原因：

- 避免打破现有价格计算 helper
- 避免影响回收站、搜索、商品卡基础展示

#### 新增层级可售配置字段

在商品记录中新增：

- `enabledConfigIdsByTier`
  - `single: string[]`
  - `double: string[]`
  - `triple: string[]`

语义：

- 每个层级下，哪些默认价格配置项对该商品是真正开放销售的
- 只有被勾选的配置项才参与该商品的价格展示与后续承接

#### 商品编辑表单结构

商品编辑改为两层：

- 第一层：勾选该商品支持哪些层级
- 第二层：对每个已勾选层级，展示对应默认价格配置项

每个层级分组内：

- 可勾选该层级下支持的价目项
- 对已勾选项填写加价
- 默认加价为 `0`

#### 不再使用全局扁平表达的内容

以下信息不再作为全局唯一真相：

- 全局规格多选决定全部层级
- 全局奶油多选决定全部层级
- 仅凭 `layers + specSizes + creamTypes` 反推商品真实可售组合

它们可以保留为汇总视图，但不再承担精确定价语义。

### 3. 老数据兼容与编辑回填

#### 老数据兼容目标

已经存在的商品没有 `enabledConfigIdsByTier` 时，不能直接失效。

#### 回填策略

进入编辑页时：

- 先读取旧商品的 `layers`
- 再结合旧商品的 `specSizes`、`creamTypes`、`priceAdjustmentsByConfigId`
- 按默认价格中心匹配出每个层级可能兼容的配置项
- 生成一份初始 `enabledConfigIdsByTier`

规则：

- 若某个 `configId` 已在 `priceAdjustmentsByConfigId` 中出现，优先视为已启用
- 若旧商品只声明了层级和扁平规格/奶油，但未显式填配置项，则按兼容项推一份初始启用集合，保证商家能继续编辑

#### 保存策略

- 新编辑后的商品保存时，写入 `enabledConfigIdsByTier`
- 同时保持旧字段可用，避免其他模块立刻崩掉

### 4. 商品卡展示与搜索摘要

商品卡继续展示：

- `specSummary`
- `layerSummary`
- `creamSummary`

但来源改为：

- 从 `enabledConfigIdsByTier` 反向汇总出该商品实际开放的规格、层级、奶油
- 而不是继续直接拼接扁平全局表单字段

这样可避免：

- 商品明明只开了部分双层组合，但卡片却把所有规格和奶油都展示出来

### 5. 对顾客端的后续影响

本轮不改顾客端，但必须记录后续改造方向。

需要在 `docs/TODO.md` 新增后续项，至少包括：

- 顾客端详情页如何展示同一商品下的单层/双层/三层可选项
- 顾客端在切换层级后，规格与价格如何联动
- 多图展示是否需要按层级关联展示
- 顾客端购物车和订单快照是否需要记录更精确的 `configId`

## 测试策略

### 自动化测试

必须覆盖：

- 首页搜索输入、清除、恢复完整列表
- 商家订单搜索输入、清除、恢复完整列表
- 商家商品搜索输入、清除、恢复完整列表
- 商品同时配置单层与双层后，保存成功
- 商品编辑回填单层/双层/三层混合配置
- 老商品无新字段时仍可回填和保存
- 商品卡汇总展示来自新层级配置

### 微信开发者工具补测

必须覆盖：

- 顾客首页搜索框清除按钮只保留一个圆形按钮
- 商家订单管理搜索框清除按钮只保留一个圆形按钮
- 商家商品管理搜索框清除按钮只保留一个圆形按钮
- 点击清除后列表立即恢复
- 商品新增页同时勾选单层和双层并分别配置价格项
- 商品编辑页打开后正确回填多个层级配置

## 风险与边界

- 商品编辑表单将从“扁平筛选”升级为“按层级选价目项”，这是结构性纠偏，不是表层样式微调
- 批量编辑本轮不扩成“按层级独立配置”，只保证不破坏现有批量能力
- 顾客端当前仍是旧承接模型，本轮只能把商家端数据先做正确并留后续 TODO，不宣称顾客端已经闭环
- 若需要修改 TDesign 样式补丁，必须同步 `miniprogram_npm` 与 `node_modules` 对应产物，避免构建 npm 后回退
